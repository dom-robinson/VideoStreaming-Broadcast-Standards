🔗 HLS RFC 8216

MPEG-DASH

DASH Protocol
Dynamic Adaptive Streaming over HTTP

ISO standard for adaptive streaming. Uses MPD manifest files and fMP4 segments. Codec agnostic and supports multiple DRM systems. Latency: 2-30s.

🔗 DASH Industry Forum
MPD Manifest
Media Presentation Description

XML-based manifest file describing available media representations, segments, and timing. Supports live and on-demand content.

🔗 ISO Standard
fMP4 Segments
Fragmented MP4

Media container format for DASH streaming. Allows efficient seeking and adaptive streaming. Compatible with CMAF standard.

🔗 ISO 14496-12

Apple HLS

HLS Protocol
HTTP Live Streaming

Apple's adaptive streaming protocol. Uses M3U8 playlists and TS or fMP4 segments. Dominant on mobile platforms. Native iOS/macOS support.

🔗 Apple Developer
M3U8 Playlist
Extended M3U Playlist

Text-based manifest format for HLS. Lists media segments with timing and quality information. Supports live and VOD content.

🔗 IETF RFC 8216
LL-HLS
Low-Latency HLS

Extension reducing HLS latency to 2-6 seconds. Uses partial segments and blocking playlist reload. Enables near real-time streaming.

🔗 Apple Documentation

CMAF Standard

CMAF Protocol
Common Media Application Format

Unified format for DASH and HLS delivery. Reduces storage costs by 75%. Single encode for multiple protocols. Supports low-latency streaming.

🔗 ISO 23000-19
Chunked Encoding
Chunked Transfer Encoding

Enables low-latency streaming by sending partial segments. Reduces glass-to-glass latency. Used in both CMAF and LL-HLS.

🔗 W3C Specification

Real-Time

WebRTC
Web Real-Time Communication

Browser-native real-time streaming. Sub-500ms latency. P2P connections or SFU scaling. DTLS security. Growing integration with traditional streaming.

🔗 WebRTC.org
WHIP/WHEP
WebRTC-HTTP Ingestion/Egress Protocol

Standardized protocols for WebRTC streaming. WHIP for ingestion, WHEP for playback. Simplifies WebRTC implementation for broadcasters.

🔗 IETF Draft

HTTP-Based

HTTP/2
HTTP/2 Protocol

Multiplexed HTTP with server push capabilities. Reduces connection overhead. Enables efficient streaming with binary framing and header compression.

🔗 IETF RFC 7540

Latest HTTP version using QUIC transport. Reduces latency with 0-RTT connection establishment. Better performance over lossy networks.

Encryption protocol securing HTTP streams. TLS 1.3 provides forward secrecy and reduced handshake latency. Required for premium content delivery.

Real-Time Protocols

RTP
Real-time Transport Protocol

Standard for audio/video delivery over IP networks. Provides sequencing, timing, and payload identification. Used in WebRTC and professional broadcasting.

🔗 IETF RFC 3550
RTCP
RTP Control Protocol

Companion protocol to RTP providing quality feedback, participant identification, and session control. Essential for adaptive real-time streaming.

🔗 IETF RFC 3550
SRTP
Secure Real-time Transport Protocol

Encrypted version of RTP providing authentication and replay protection. Used in WebRTC and secure professional streaming applications.

🔗 IETF RFC 3711

Professional Transport

SRT Protocol
Secure Reliable Transport

Open-source protocol for low-latency streaming over unpredictable networks. 120ms+ latency, AES encryption, caller/listener mode. Widely adopted for contribution.

🔗 SRT Alliance
RIST Protocol
Reliable Internet Stream Transport

Professional streaming protocol with three profiles. Based on RTP with advanced error correction. Designed for broadcast contribution over public internet.

🔗 RIST Forum
SMPTE ST 2110
Professional Media Over IP

Suite of standards for uncompressed media over IP networks. Separate essence flows for video, audio, and metadata. Requires PTP synchronization.

🔗 SMPTE Standards

Fast, connectionless protocol. Low overhead, no guaranteed delivery. Used for real-time streaming where speed is more important than reliability.

Reliable, connection-oriented protocol. Guaranteed delivery with error correction. Used for HTTP streaming, file transfers, and applications requiring reliability.

Network layer protocols for packet routing. IPv6 provides larger address space and better support for real-time services. Gradual transition ongoing.

Standard Workflow Paths

🔴 Live Streaming Workflow

Camera → 12G-SDI → Video Encoder → [H.264/HEVC Codec] → fMP4 Container → SRT Transport → CDN Ingest → DASH/HLS ABR → Player → Display Device

📺 Professional Broadcast Workflow

Studio Camera → 12G-SDI → Production Switcher → ST 2110 IP Network → Master Control → MPEG-TS → DVB/ATSC Modulation → Transmitter → Set-Top Box → TV

🌐 OTT Streaming Workflow

Content Source → Cloud Encoder → [Multi-Codec ABR] → CMAF Packaging → CDN Distribution → Adaptive Streaming (DASH/HLS) → Smart TV/Mobile → Display

🎮 Interactive/Real-Time Workflow

Game/Application → Screen Capture → [Low-Latency Codec] → WebRTC P2P/SFU → Browser/App → Real-Time Display (<500ms latency)
